⏺ Analisi dell'errore — IusMente

  Flusso dell'errore

  Il messaggio "Si è verificato un errore. Riprova tra qualche istante." compare
  in app/page.js:116, dentro al blocco catch di handleInvia. Viene mostrato
  quando data.error è valorizzato (cioè il backend risponde con un errore)
  oppure quando fetch/res.json() falliscono.

  A sua volta, il backend in app/api/chat/route.js:127-133 intercetta qualsiasi
  eccezione e restituisce sempre:
  { "error": "Errore durante la generazione della risposta." }
  Il console.error lato server cattura la causa vera, ma il client non la riceve
  mai.

  ---
  Cause probabili (in ordine di verosimiglianza)

  1. ❌ Manca la variabile d'ambiente GEMINI_API_KEY

  Nella root del progetto non esiste alcun .env, .env.local o .env.production.
  La ls ha confermato solo .gitignore, .git, .next, app, node_modules, ecc.

  route.js:4 esegue:
  const genAI = new GoogleGenerativeAI(process.env.GEMINI_API_KEY)
  Senza la chiave, ogni chiamata a generateContent fallisce con errore di
  autenticazione → 500 dal backend → "errore" lato UI.

  2. ❌ Modello gemini-1.5-flash ritirato

  route.js:71 usa model: "gemini-1.5-flash". Google ha ritirato Gemini 1.5 Flash
  ad aprile 2025, e oggi (giugno 2026) quel nome restituisce 404 Not Found
  dall'API di Gemini, anche con chiave valida. È la causa più probabile se la
  chiave API è già configurata sul deploy.

  3. ⚠️ SDK errato

  In package.json ci sono due SDK Gemini:
  "@google/genai": "^2.9.0",          // nuovo
  "@google/generative-ai": "^0.24.1"  // vecchio, usato da route.js
  Il nuovo SDK @google/genai usa nomi modello diversi (gemini-2.5-flash,
  gemini-2.0-flash) e un'API diversa. Il file route.js usa ancora il vecchio
  SDK, che è in manutenzione.

⏺ Per la versione da package.json, devo esporla al client. Il modo più semplice
  in Next.js è passarla come env con prefisso NEXT_PUBLIC_. Aggiorno
  package.json con un campo version (Next.js lo legge automaticamente tramite
  process.env.NEXT_PUBLIC_APP_VERSION se configurato, oppure lo leggo da
  package.json server-side e lo passo come prop).

  Il modo più pulito: scrivo un piccolo file app/version.js che esporta la
  versione letta da package.json server-side, e la importo nel client. Però in
  Next.js il client non può importare direttamente JSON. Soluzione: passo la
  versione come prop da layout.js (server component) a page.js (client
  component). Ma page.js è Home definito direttamente, senza wrapper.
  Alternativa più semplice: uso una variabile NEXT_PUBLIC_APP_VERSION che leggo
  da package.json con uno script, oppure hardcodo e basta.

  Vista la semplicità richiesta, faccio così: leggo la versione una volta
  server-side in layout.js e la passo via <meta> o via Context. La cosa più
  rapida e funzionante: la leggo server-side e la metto in un <script 
  id="app-version"> che il client legge. Troppo complicato.

  Soluzione finale, semplice e robusta: creo un file lib/version.js server-side
  che esporta la versione letta da package.json, e la importo in app/page.js
  come import lato server (Next.js lo consente se il file non usa fs
  direttamente nel browser). Però se uso import version from '../package.json',
  funziona in Next.js perché risolve a build time.
